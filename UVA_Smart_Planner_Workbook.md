# 1. Project Summary and Scope

UVA Smart Planner is a single-user web-based planning agent designed for UVA students who need help turning scattered assignments, class schedules, and routine constraints into a stable and executable daily plan. The system reads an uploaded ICS course calendar, extracts tasks from uploaded syllabus files (PDF or Word), allows manual edits, and then generates a time-blocked daily schedule. When a new conflict appears, such as an added meeting or a task update, the system automatically replans the remaining part of the day while preserving hard constraints and minimizing unnecessary disruption.

This workbook consolidates the current proposal into a revised implementation-oriented version. The final scheduler is CP-SAT rather than a greedy heuristic, and the model-access layer is designed around Codex CLI OAuth credential reuse in the style documented by OpenClaw.

## Scope assumptions used throughout this document

- Single-user web application built with Django.

- Inputs are limited to uploaded ICS calendars, uploaded PDF/DOCX syllabi, and manual task/routine entry.

- No Canvas or SIS integration is included in this version.

- The final scheduling engine is OR-Tools CP-SAT.

- The model layer is used for syllabus task extraction, natural-language patch parsing, and summary generation; it is not used as the final scheduler.

## Core capabilities

- Initial schedule generation: build a feasible time-blocked plan from tasks, deadlines, classes, and routines.

- Dynamic replanning: detect conflicts and re-solve the remaining schedule from the current time onward.

- Deadline confidence: estimate whether the user remains on track before upcoming deadlines.

- Natural-language updates: translate user updates into structured patches and summarize the resulting changes.

# 2. Technical Proposal

## 2.1 Motivation and Problem Definition

UVA students often manage multiple courses, assignments, meetings, and extracurricular activities simultaneously. Task information is scattered across syllabi, email, announcements, and personal notes, while daily schedules are frequently disrupted by last-minute meetings, office hours, or workload changes. Existing calendar and to-do tools mainly act as reminders; they do not reason about real constraints such as class times, sleep windows, workload limits, and task priorities, nor do they intelligently replan when the environment changes. The result is a gap between knowing what needs to be done and maintaining a realistic, executable daily plan.

The proposed system, UVA Smart Planner, addresses this problem as a constraint-aware planning agent. It accepts a structured set of tasks, a course calendar, and user routines/preferences, then produces a feasible time-blocked daily schedule. When new conflicts appear, it automatically recomputes a revised plan while preserving hard constraints and minimizing unnecessary movement of existing blocks.

### Technical specification

- Inputs: uploaded ICS course calendar, uploaded syllabus files (PDF/DOCX), manually entered tasks, manually entered daily routines, and natural-language updates from the user.

- Outputs: time-blocked daily schedule, weekly candidate task allocation, conflict alerts, deadline confidence indicators, and natural-language update summaries.

- Planning focus: daily schedule is the primary optimization target; weekly planning is used mainly to filter and rank candidate tasks.

## 2.2 Methods

The project uses a hybrid architecture: a model-assisted understanding layer plus a local optimization-based scheduling layer. The model converts unstructured syllabus text and natural-language updates into structured data; the final schedule is generated by a CP-SAT solver rather than by the model itself.

### 2.2.1 Task extraction with Codex

Uploaded syllabus files are converted into raw text and then split into chunks. A Codex-backed extraction service processes each chunk and returns structured task candidates in strict JSON format. Each candidate task includes title, course name, due datetime, estimated duration, category, confidence score, and raw excerpt. Low-confidence extractions are surfaced on a review page so the user can correct due dates, durations, or task labels before the tasks are committed to the database.

### 2.2.2 Candidate task selection

The system does not attempt to schedule every task at once. Instead, it deterministically selects a daily or weekly candidate set using task status, deadline proximity, carry-over status, and priority. This keeps the scheduler focused, explainable, and efficient.

### 2.2.3 Final scheduling algorithm: CP-SAT

The final scheduling algorithm is a constraint optimization model implemented with Google OR-Tools CP-SAT. Time is discretized into fixed slots (for example, 30-minute slots), and the solver decides how to allocate task time across the available slots. CP-SAT is preferred over a greedy heuristic because it reasons globally across interacting constraints and supports optimization over multiple soft objectives.

### Hard constraints enforced by the scheduler

- No overlap with fixed ICS events such as classes and manually added fixed meetings.

- No overlap between simultaneously assigned task blocks.

- Preservation of sleep windows and other non-negotiable routine blocks.

- Respect for maximum continuous work duration, with breaks inserted as needed.

- Task-duration coverage: each scheduled task must receive enough total time to satisfy its required duration.

### Soft objectives optimized by the scheduler

- Schedule urgent and high-priority tasks earlier.

- Increase buffer before task deadlines whenever possible.

- Reduce fragmentation by minimizing unnecessary task splitting.

- Align work blocks with preferred study windows.

- During replanning, minimize movement of future schedule blocks.

### 2.2.4 Dynamic replanning

When the user adds a new event, marks a task as complete, or changes a task estimate, the system does not blindly rebuild the entire day. Instead, it uses rolling-horizon replanning: past schedule blocks are locked, optionally very near-future blocks are frozen, and only the remaining future portion of the day is re-optimized. A movement penalty in the objective discourages unnecessary reshuffling and preserves stability.

### 2.2.5 Natural-language summaries

After scheduling or replanning, the model generates a short summary describing what changed, what stayed fixed, and whether the user remains on track before upcoming deadlines. A fallback template-based summary is also provided for robustness.

## 2.3 Authentication and model integration

The project uses Codex CLI OAuth rather than hard-coded API keys as the primary model-authentication path. The implementation follows the OpenClaw-style reuse strategy: the system reads locally available Codex CLI credentials from the standard auth location, mirrors necessary credential metadata into an application-managed cache, and uses a dedicated provider layer to validate, refresh, and serve those credentials to higher-level extraction and summary services.

- The Django application does not implement a full browser-based end-user OAuth flow in this version.

- The intended workflow is: run `codex --login` once on the host machine, then let the app reuse and manage those credentials locally.

- The provider layer must validate credential freshness, handle refresh when necessary, and isolate authentication logic from the rest of the app.

## 2.4 Data and libraries

- Framework: Django monolith, optionally with Django REST Framework for internal JSON endpoints.

- Calendar parsing: icalendar.

- PDF extraction: pdfplumber.

- Word extraction: python-docx.

- Scheduling engine: Google OR-Tools CP-SAT.

- Frontend schedule rendering: FullCalendar.

- Asynchronous partial UI updates: HTMX.

## 2.5 Measure of success

### Primary evaluation metrics

- Feasibility: number of hard-constraint violations (target: zero).

- Deadline confidence: remaining buffer before due dates and whether tasks remain on track.

- Schedule stability: number of blocks moved during replanning.

- Fragmentation: number of task splits and average continuous focus block length.

- Weighted completion score based on task priority.

### Baselines

- To-do list only (no time-blocking).

- Earliest-deadline-first heuristic.

- Generate once and do not replan after conflicts occur.

# 3. System Architecture

The system is organized as a Django monolith with clear internal layers. This keeps deployment simple while preserving clean module boundaries.

| Layer | Main Responsibility | Key Components |
| --- | --- | --- |
| UI layer | Render pages and partial updates | Django templates, HTMX, FullCalendar |
| Web layer | Handle routes, forms, uploads, and user actions | views.py, forms.py, urls.py |
| Service layer | Business workflows and external integrations | ICS parsing, syllabus text extraction, Codex provider |
| Scheduling layer | Optimization, replanning, metrics | scheduler_cp_sat.py, replanner.py, metrics.py |
| Persistence layer | Save structured state | SQLite models + local JSON/file snapshots |
| Runtime resources | Uploads, cached text, OAuth cache, logs | media/, data/, logs/, ~/.codex/auth.json |

## Architecture flow

1. ICS upload -> calendar parsing -> CalendarEvent records

1. Syllabus upload -> text extraction -> chunking -> Codex extraction -> Task records

1. Task selection + profile + fixed events -> CP-SAT scheduler -> ScheduleBlock records

1. Natural-language update -> patch parsing -> replanner -> new schedule version + summary

# 7. Local Data Storage Specification

The system stores state both in SQLite and in local files. Database tables hold normalized operational state, while the file system stores uploads, extracted text, versioned snapshots, metrics, and OAuth mirrors.

## 7.1 Database tables

- UserProfile — routines and study preferences.

- CalendarEvent — imported ICS events and manual fixed events.

- Task — extracted or manually entered tasks.

- ScheduleBlock — generated schedule blocks per version.

- ReplanLog — trigger, version diff, and summary for every replan event.

## 7.2 File-system storage

- media/ics/ — uploaded ICS files.

- media/syllabi/ — uploaded PDF/DOCX syllabus files.

- media/extracted_text/ — raw extracted text and chunk metadata.

- media/task_json/ — intermediate or exported task JSON files.

- data/oauth_cache/ — mirrored OAuth metadata and refresh lock files.

- data/schedule_versions/ — versioned JSON schedule snapshots.

- data/conflict_reports/ — infeasibility reports.

- data/metrics_exports/ — evaluation outputs.

- data/debug_snapshots/ — solver input/output snapshots for debugging.

- logs/ — application, scheduler, model, and OAuth logs.
